1. Solution Overview

The Hydrocarbon Tank Stock Reconciliation solution is a full-stack enterprise application built on SAP BTP that automates the daily reconciliation of physical tank measurements (ATG dip readings from IS-OIL HPM) against SAP book stock. It classifies variances, routes urgent ones through a supervisor approval workflow, and posts approved goods movements back to SAP via IS-OIL Physical Inventory BAPIs.

App URL: https://tank-reconciliation-approuter-proud-jackal-qo.cfapps.us10.hana.ondemand.com

Key Value Proposition

  • Eliminates manual spreadsheet-based reconciliation
  • Provides real-time variance classification with GREEN / AMBER / RED thresholds
  • Enforces IS-OIL compliant goods movement postings via Physical Inventory BAPIs
  • Full audit trail from data ingestion to posting (M1–M6)
  • AI-powered assistant for operator guidance and decision support
  • Role-based access control via SAP XSUAA

2. Business Problem

Oil terminals operate large storage tanks containing hydrocarbons (diesel, propane, gasoline). Every day, physical dip readings are taken from Automatic Tank Gauges (ATG) and recorded in the IS-OIL system (OIB_TANKDIP table). These physical measurements must be compared against the SAP book stock (RELSTOCK) to identify losses, gains, or measurement errors.

Challenges Before This Solution

  • Manual comparison between ATG readings and SAP book stock
  • No automated threshold-based classification
  • No structured approval workflow for large variances
  • No audit trail for regulatory compliance
  • Excise duty implications for taxed hydrocarbons not handled automatically
  • No AI-assisted decision support for operators

3. Functional Design

3.1 Reconciliation Flow

  1. Admin triggers a reconciliation run for a selected date
  2. System fetches the latest dip reading from IS-OIL (OIB_TANKDIP) for each configured tank via ZTANK_DIP_SRV_SRV
  3. Physical quantity (QUAN_SKU) is compared against book stock (RELSTOCK)
  4. Variance is calculated and classified as GREEN / AMBER / RED
  5. GREEN and AMBER variances are auto-posted; RED variances are held for supervisor approval
  6. Approved postings trigger IS-OIL Physical Inventory BAPIs to post goods movement
  7. Full audit trail written at each milestone (M1–M6)
  8. Notifications sent via MS Teams webhook or BTP Alert Notification Service

3.2 Variance Classification

ClassificationThresholdAction
🟢 GREENDelta ≤ 0.5%Auto-posted immediately
🟡 AMBER0.5% < Delta ≤ 2.0%Auto-posted, flagged for monitoring
🔴 REDDelta > 2.0%Held — requires supervisor approval

Thresholds are configurable per tank in TankConfiguration.

3.3 Approval Workflow

  1. RED-classified tanks appear in the Approval Queue
  2. Supervisor selects a Reason Code and enters a comment
  3. Supervisor clicks Approve or Reject
  4. Approved tanks trigger M5 goods movement posting
  5. Rejected tanks are recorded with reason in the audit trail

IS-OIL Reason Codes

CodeDescription
01Measurement
02Transport Gain
03Transport Losses
04Customer not available
05Insufficient quantity delivered
06Lost quantity

4. Milestones M1–M6

MilestoneNameDescriptionStatus
M1 Data Ingestion Fetch latest dip from OIB_TANKDIP via ZTANK_DIP_SRV_SRV. Non-standard input detection: zero/negative qty, qty >500k TO, book stock ≤0, variance >1000%. ✅ Complete
M2 VCF Correction Volume Correction Factor applied. QUAN_SKU in OIB_TANKDIP is already VCF-corrected by IS-OIL — factor 1.0 is correct. VCF API URL supported if external correction needed. ✅ Complete
M3 Variance Calculation Delta = netVolumePhysical − bookStock. DeltaPercent = |delta/bookStock| × 100. Classification: GREEN/AMBER/RED per tank tolerances. ✅ Complete
M4 Approval Workflow RED tanks routed to Approval Queue. Supervisor approves/rejects with reason code [01–06] and comment. High variance warning (>1000%) written to audit trail. License constraint check (inactive tanks blocked). ✅ Complete
M5 Goods Movement Posting IS-OIL Physical Inventory BAPIs: BAPI_MATPHYSINV_CREATE → BAPI_MATPHYSINV_COUNT → BAPI_MATPHYSINV_POSTDIFF. Posts movement type 701 (gain) or 702 (loss). Dip timestamp used as count date. Material document stored and displayed. ✅ Complete
M6 Alerts & Notifications MS Teams webhook notification with GREEN/AMBER/RED status, tank counts, and run summary. BTP Alert Notification Service supported. In-app notification bell with badge count. ✅ Complete

6. IS-OIL Integration

6.1 Tank Dip Data Source

Physical inventory data is sourced from the IS-OIL HPM table OIB_TANKDIP on OGS/650:

FieldMeaning
SOCNRTank ID (20-char, zero-padded)
QUAN_SKUPhysical quantity in TO (metric tons) — already VCF-corrected by IS-OIL
RELSTOCKBook stock (SAP stock at time of dip)
ETMSTMTimestamp of dip reading
MEINSUnit of measure (TO)
QUAN_LVCLine Volume Corrected (different UoM — not used for reconciliation)
Important: QUAN_SKU is already VCF (Volume Correction Factor) corrected by IS-OIL before writing to OIB_TANKDIP. The reconciliation uses factor 1.0 — no external VCF API is needed.

6.2 Plant / Storage Location Mapping

Tank SOCNR is mapped to SAP Plant and Storage Location via OIISOCISL table on OGS/650.

6.3 M5 Goods Movement Posting — IS-OIL BAPI Chain

The posting flow uses three standard IS-OIL Physical Inventory BAPIs:

Step 1: BAPI_MATPHYSINV_CREATE
        → Creates a new Physical Inventory document (IKPF)
        → Parameters: Plant, Storage Location, Doc Date, Material, Batch

Step 2: BAPI_MATPHYSINV_COUNT
        → Records the physical count quantity
        → Parameters: PI Doc, Fiscal Year, Count Date, Item, Material, Batch, Qty, UoM

Step 3: BAPI_MATPHYSINV_POSTDIFF
        → Posts the difference as goods movement (701 gain / 702 loss)
        → IS-OIL validates: OIH10 → OIH01 → OIH07
        → BAPI_TRANSACTION_COMMIT after each step

8. Intelligence Features

8.1 Non-Standard Input Detection (M1)

ConditionAudit Message
Physical qty ≤ 0INVALID: Physical quantity is zero or negative
Physical qty > 500,000 TOSUSPICIOUS: Physical quantity exceeds 500,000 TO
Book stock ≤ 0WARNING: Book stock is zero or negative
Variance > 1000%DATA_QUALITY: Variance X% exceeds 1000%

8.2 License Constraint Check (M4)

  • Inactive tanks (active = false in TankConfiguration) → blocked from posting with HTTP 403
  • Variance >1000% → HIGH VARIANCE WARNING written to M4 audit entry before allowing approval

8.3 AI Chat Assistant

  • Powered by SAP AI Core with GPT-4o model
  • System prompt injects live reconciliation context (latest run, tank results, pending approvals)
  • Fallback rule-based replies if AI Core is unavailable
  • Supports queries: latest run status, flagged tanks, approval recommendations, tank-specific variance analysis
  • Specific recommendation logic: data-quality warnings for >500% variance, O4_TIGER re-measurement guidance

8.4 VCF (Volume Correction Factor)

  • QUAN_SKU in OIB_TANKDIP is already VCF-corrected by IS-OIL HPM
  • Factor 1.0 applied (no double-correction)
  • External VCF API supported via VCF_API_URL env var if gross volume correction needed in future
